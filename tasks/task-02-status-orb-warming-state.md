# Task 02: Состояние WARMING в статус-орбе

**Story:** story-voice-trigger-warmup.md
**Роль-исполнитель:** Claude Code
**Приоритет:** высокий
**Зависимости:** task-01 (нужна функция `warm_up_model()` для подключения к
переходу состояния)

## Описание

Добавить состояние `WARMING` в state machine статус-орба — между `IDLE` и
`LISTENING`. Согласуется с будущей Status Console (пять состояний + Error),
так что здесь нужно решить: это шестое состояние или под-состояние
существующего — финальное решение стоит принять при ревью Status Console
целиком, а не в этой задаче изолированно.

Переход: любой триггер (хоткей / клик по орбу / будущий wake word) →
`IDLE → WARMING`, параллельно стартует `warm_up_model()` и захват речи через
VAD. Переход `WARMING → LISTENING` происходит по готовности модели (или по
таймауту ожидания — см. критерии приёмки).

## Реализация (набросок)

- Визуально: `WARMING` можно делать теплым/amber-adjacent, но цвет и анимация
  должны четко отличаться от data-locality/cloud warning. Пользователь не
  должен читать прогрев локальной модели как «данные ушли наружу».
- В UI есть две независимые privacy axes: data locality (локальный backend /
  внешний provider) и system visibility mode (например, Open / Hidden). `WARMING`
  относится к runtime state, а не к privacy mode.
- Аудио/реплика пользователя, сказанная во время `WARMING`, должна
  буферизоваться VAD-пайплайном, а не отбрасываться и не отправляться в
  Ollama до подтверждения готовности модели.
- Таймаут ожидания: если прогрев занимает дольше ожидаемого (например, дольше
  времени, замеренного Андреем в task-01), орб уходит в `Error` с понятным
  сообщением в «Событиях системы», а не зависает в `WARMING` бесконечно.

## Критерии приёмки

- [ ] `WARMING` визуально отличим от `LISTENING`, `IDLE`, `Error` и от
      amber cloud-индикатора.
- [ ] Речь, сказанная во время `WARMING`, не теряется — буферизуется и
      отправляется после готовности модели.
- [ ] Есть верхняя граница ожидания в `WARMING` с переходом в `Error` при
      превышении.
- [ ] Состояние логируется в «События системы» (вход/выход, длительность).

## Stop condition / hardware handoff

Анимация и тайминги требуют визуальной проверки на реальном экране Андрея
(субъективное ощущение «не мигает ли слишком резко», «не сливается ли с
cloud-индикатором»). Claude Code реализует и останавливается на этапе
готового прототипа состояния — визуальное ревью и тайминг-тюнинг передаются
Андрею.
