# Story: Активация голосового канала — триггер + прогрев для рваного режима работы

**Статус:** Backlog (кандидат после v1.1.1)
**Родитель:** activation / Status Console roadmap (Jarvis Local)
**Связано с:** будущая Status Console (новое состояние `WARMING` в статус-орбе)
**Дата создания:** 2026-07-06

## Контекст

Сейчас голосовой канал работает в режиме «всё включено / всё выключено», что удобно
для непрерывной работы, но плохо подходит для рваного режима использования —
коротких обращений с большими паузами между ними. За время паузы Ollama по
таймауту (`keep_alive`, по умолчанию 5 минут) выгружает модель из VRAM, и первое
обращение после паузы получает штраф на холодный старт (загрузка Gemma 4 12B QAT
в 16GB VRAM — заметное время).

Нужен механизм, который:
1. Даёт пользователю явный (или полуявный) способ сообщить системе «я хочу
   говорить прямо сейчас»;
2. Параллельно с этим триггерит прогрев Ollama, чтобы к моменту завершения
   фразы пользователя модель уже была готова принять запрос.

## Цель

Сократить ощутимую задержку первого ответа после паузы, не жертвуя приватностью
и не усложняя основной voice pipeline.

## Ключевые решения

- Триггер и прогрев — два разных механизма, соединённые через общий вызов
  `trigger_warmup()`. Не смешивать в одну функцию.
- Добавляется новое состояние `WARMING` в state machine статус-орба (ложится в
  будущую Status Console — между `IDLE` и `LISTENING`).
- Источники триггера, которые можно комбинировать:
  1. Push-to-Talk хоткей через общий `HotkeyProvider`. Windows-реализация
     должна использовать `RegisterHotKey`, не keystroke-sniffing библиотеку
     `keyboard`. Миграция существующих hotkeys на этот provider вынесена в
     отдельную story, чтобы не вводить две долговременные модели hotkey
     handling.
  2. Клик по статус-орбу — универсальный fallback, не зависит от OS-level
     global hotkey APIs.
  3. Voice wake word через `openWakeWord` (готовая модель «hey jarvis») —
     вынесено в отдельный backlog-item, см. «Вне рамок».
- Независимо от триггеров: увеличение `keep_alive` в запросах к Ollama снижает
  саму частоту необходимости прогрева.

## Критерии готовности (Definition of Done)

- [ ] Прогрев Ollama можно вызвать программно (`warm_up_model()`), асинхронно,
      не блокируя UI-поток.
- [ ] Есть минимум один рабочий триггер на Windows (хоткей и/или клик по орбу).
- [ ] Статус-орб визуально отражает состояние `WARMING` отдельно от `LISTENING`.
- [ ] `keep_alive` вынесен в конфиг, а не захардкожен.
- [ ] `day0_checks.py` покрывает: срабатывание хоткея, срабатывание клика по
      орбу, замер времени прогрева модели.

## Вне рамок этой story

- **Wake word** (`openWakeWord`, модель «hey jarvis») — отдельная story.
  Требует предварительного day-0-style эксперимента по false positive rate в
  реальных условиях Андрея, аналогично Thinking mode toggle. Не блокирует эту
  story.
- **Полная миграция всех текущих хоткеев с `keyboard` на `HotkeyProvider`** —
  отдельная story. PTT может быть первым потребителем нового provider, но
  production-состояние не должно надолго оставлять часть hotkeys на `keyboard`,
  а часть на WinAPI.
- **Linux/Wayland-реализация хоткея** — заблокирована внешней проблемой:
  портал `org.freedesktop.portal.GlobalShortcuts` на GNOME отклоняет привязки
  для не-Flatpak приложений (актуально на начало 2026). Абстракция
  `HotkeyProvider` закладывается сейчас (см. task-03), конкретная
  Linux-реализация — отдельная будущая задача.

## Открытые вопросы

- Какую длину паузы считать границей «рваного режима» — 5 минут? 15? Это
  влияет на выбор дефолтного значения `keep_alive`. Требует эмпирической
  проверки Андреем на реальном использовании.
- Нужен ли короткий звуковой/визуальный ack на срабатывание триггера (до того,
  как VAD начнёт захват речи), чтобы пользователь понимал, что система его
  услышала?
